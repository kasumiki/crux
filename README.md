# Crux

### Get to the crux of every command.

**Crux is a context compiler for AI coding assistants.** It compresses the noisy CLI output
your agent reads — `git diff`, `pytest`, `npm install`, `terraform plan`, `kubectl`, `docker` —
down to what matters, so you spend **up to 99% fewer tokens** while every error, diff, and stack
trace is *guaranteed* to survive.

## Why Crux

Every command your AI assistant runs burns tokens — and most of that output is noise. A
500-line `git diff`, a `pytest` run with 200 passing tests, an `npm install` with 80 packages:
the model only needs the errors, the changed files, and the result. The rest is wasted context
and wasted money.

Most tools that tackle this are a bag of per-tool truncators with fixed line limits. **Crux is
different.** It adds one general, principled model on top:

- 🎯 **Importance-ranked.** Every line is scored `CRITICAL` (errors, tracebacks) → `HIGH`
  (warnings, diffs) → `MEDIUM` → `LOW` → `NOISE` (progress bars, spinners) — independent of
  the tool.
- 🪙 **Budget-aware.** Set a token budget and Crux compresses *any* output to fit it — even
  from a tool it has never seen.
- 🛡️ **Safe by guarantee.** `CRITICAL` lines are *never* dropped, even if that exceeds the
  budget. You never lose an error message.
- ⚡ **Deterministic & offline.** Pure parsing — no LLM calls, no network, ~0 ms added latency.
- 🔌 **Install once, forget it.** Works automatically in the background for **Claude Code** and
  **Antigravity CLI**.
- 🔍 **Transparent.** Every reduction reports what it elided, so nothing vanishes silently.

## Quick start

```bash
# 1. Add and install the plugin in Claude Code
/plugin marketplace add kasumiki/crux
/plugin install crux

# 2. Restart Claude Code. That's it — Crux now compresses CLI output automatically.

# 3. (optional) Confirm it's working
crux doctor          # health check: install, hooks, data dir, config
crux stats           # tokens saved this session and over time
```

No configuration is required. Crux begins compressing the moment it's installed.

## Installation

**Prerequisites:** Python 3.10+ and Claude Code and/or Antigravity CLI.

### Claude Code plugin (recommended)

```bash
/plugin marketplace add kasumiki/crux
/plugin install crux
```

### Manual installer (Claude Code and/or Antigravity)

```bash
git clone https://github.com/kasumiki/crux && cd crux
python3 install.py --target both     # or: claude | antigravity
crux doctor                          # verify
```

Uninstall any time with `python3 install.py --uninstall` (add `--keep-data` to keep your stats).

### pip (just the CLI)

```bash
pip install crux-cli                 # provides the `crux` command
```

The plugin install and the pip install are independent — the plugin ships a self-contained
copy, so they never conflict.

## How it works

```
command ──▶ gate ──▶ processor (tool-aware shaping) ──▶ reducer (importance + budget) ──▶ output
```

1. **Gate** — decides whether a command is even eligible. Interactive tools (`vim`, `ssh`),
   `sudo`, output redirections, and complex pipelines are skipped untouched.
2. **Processor** — 30+ specialized processors give tool-aware shaping (it knows what a
   `git diff` or a `pytest` run looks like). The first whose pattern matches wins; a `generic`
   processor handles everything else.
3. **Reducer** — the final authority. It ranks every line by [importance](#why-crux) and
   compresses to your token budget, dropping noise first and **guaranteeing errors survive** —
   on *any* output, even tools Crux has never seen. It appends a one-line footer noting what
   was elided.

**Claude Code** integrates through a PreToolUse hook that transparently runs the command through
a wrapper (Claude can't edit output after the fact). **Antigravity** replaces output directly via
its AfterTool hook. Both share the same gate, processors, and reducer.

**Supported tools:** git · GitHub CLI · docker · kubectl · helm · terraform · pulumi · cdktf ·
ansible · cargo · go · npm/pip/poetry/uv · test runners (pytest, jest, vitest, go/cargo test) ·
linters (eslint, ruff, pylint, clippy, mypy) · build tools · cloud CLIs (aws, gcloud, az) ·
database clients · curl/wget · grep/rg/fd · ls/find/tree · cat/head/tail · env · journalctl ·
ssh · jq/yq · nix · mise · just · and a generic fallback for everything else.

## CLI reference

```bash
crux stats                  # token savings — this session and lifetime
crux benchmark "git diff"   # measure compression on a command
crux explain "git log"      # show how a command is routed (or why it's skipped)
crux doctor                 # diagnose install, hooks, data dir, and config
crux config show            # effective config + where each value came from
crux config get <key>       # read one setting
crux config set <key> <val> # change a setting (writes ~/.crux/config.json)
crux init-processor <name>  # scaffold a custom processor
crux update                 # check for and apply updates
crux version                # print the version
```

## Configuration

No setup is needed — Crux ships sensible defaults. When you want to tune it, settings resolve
in this order (highest priority first):

**environment variable → project file → global file → profile → default**

| Source | Where |
|--------|-------|
| Global config | `~/.crux/config.json` |
| Per-project config | `.crux.json` (searched upward from the working directory) |
| Environment variable | `CRUX_<KEY>` (e.g. `CRUX_MAX_OUTPUT_TOKENS=1500`) |

```bash
crux config show                       # see every setting and its source
crux config set max_log_entries 5
crux config set profile aggressive
```

### Profiles

One knob bundles the aggressiveness settings:

| Profile | Behavior |
|---------|----------|
| `conservative` | Compress less, keep more. |
| `balanced` *(default)* | A sensible middle ground. |
| `aggressive` | Compress harder; also caps output at a 1,500-token budget. |

### Token budget

Cap *any* command's output in tokens — the reducer keeps the most important lines and always
preserves errors:

```bash
crux config set max_output_tokens 1500   # 0 = off (default)
```

### Turning it off

- For one command: append `# crux:raw` to it.
- For a whole session: set `CRUX_BYPASS=1`.
- For one tool: `crux config set disabled_processors docker` (comma-separate for several).
- Debugging: `CRUX_DEBUG=true` logs to `~/.crux/hook.log`.

## Custom processors

Need a tool Crux doesn't shape yet? Add one — it's a single file:

```bash
crux init-processor mytool             # writes ~/.crux/processors/mytool.py
```

Edit the generated file (it sets a `priority` and `hook_patterns` and implements `can_handle()`
and `process()`), and it's auto-discovered on the next command. See
[examples/custom_processor/](examples/custom_processor/) for a worked example.

## FAQ & troubleshooting

**Is my code or output sent anywhere?**
No. Crux is pure local parsing — no LLM calls, no network, nothing leaves your machine.

**Could it hide an error from me?**
No. Errors and stack traces are classified `CRITICAL` and are never dropped — that's a hard
guarantee, verified by the test suite.

**Doesn't prompt caching already make tool output cheap?**
No — caching and Crux solve different halves of the problem. Caching makes *re-reading* an
unchanged prefix cheap; it does nothing for the cost of writing new output into the context in
the first place, and large outputs still consume the cache window and the model's context budget
on every later turn. Crux shrinks the output *before* it enters context, so the cache write is
smaller, every cached re-read is smaller, the output falls out of the cache window less often, and
you leave more room for actual work. They compose — Crux makes the thing you cache smaller.

**It doesn't seem to be compressing.**
Run `crux doctor` and address anything it flags. Then check a specific command with
`crux explain "<command>"` — it tells you whether the command is eligible and which processor
handles it. Commands with `sudo`, redirections (`>`), or complex pipes are intentionally skipped.

**How do I see how much it's saving?**
`crux stats` (per session and lifetime), or `crux benchmark "<command>"` for a single command.

**Does it slow anything down?**
No. Compression is regex/parsing only — effectively zero added latency, and it never makes an
LLM call.

**Will it work with a tool you don't list?**
Yes. Unknown tools fall through to the generic processor and the budget reducer, which compress
by importance regardless of the tool.

## Security

Crux runs locally, never transmits output, and fails open (if compression ever errors, your
original output passes through untouched). To report a vulnerability, see [SECURITY.md](SECURITY.md).
